Quickstart — 4 lines, evaluate first

copy
# pip install openai chromadb flashrank   •   export OPENAI_API_KEY=sk-...
from rag import RAG

r = RAG().add(open("docs.txt").read().split("\n\n"))
print(r.evaluate(k=5))          # ← start here: hit-rate@k (=recall@k), MRR@k, failures
print(r.answer("your question"))

evaluate() generates a question from each chunk and checks whether that chunk comes back for its own question. Low recall@k? Your retrieval is broken and no prompt-tuning will save you. Read result["failures"] — those chunk ids are candidates to inspect.

Read this before you trust the number
  • It reports hit-rate@k (= recall@k with one gold chunk per question) and MRR@k on synthetic queries — a retrieval sanity check and regression signal, not a production score.
  • Questions are written from the chunk, so they share its wording. Real user queries are harder — treat this as an optimistic upper bound.
  • Chunk overlap causes false misses (a neighbor answers the question). Inspect failures.
  • It says nothing about generation faithfulness or multi-hop questions — add an LLM-judge for those.

The real workflow: measure on a held-out set of real user queries, read the failures, change one thing, re-measure.

Build a real benchmark

evaluate() is a synthetic warm-up. The number you actually report comes from a benchmark — and a benchmark is not a golden dataset. The golden dataset is the labeled data; the benchmark is that data plus metrics plus a repeatable harness.

1 · Collect real queries
From logs, tickets, or an expert. Cold start: LLM drafts, a human edits and keeps the good ones.
2 · Label by content
Record a snippet a correct chunk must contain — not a chunk id. Survives re-chunking.
3 · Cover + freeze
Easy, hard, multi-hop, some unanswerable. ~50–100 to start. Version it; grow it from real failures.

Store the golden dataset as JSONL

# golden.jsonl — one real, labeled query per line

{"query": "which organelle makes energy in cells?", "relevant": "powerhouse of the cell"}
{"query": "how high is Everest?", "relevant": "8,849 meters"}

Run the benchmark

copy
import json
golden = [json.loads(l) for l in open("golden.jsonl")]

print(r.score(golden, k=5))                # recall@k + MRR@k over REAL queries
print(r.score(golden, k=5, rerank=True))   # does rerank help on YOUR data?
# result["misses"] = the queries that failed — your to-do list

A hit counts when any retrieved chunk contains a labeled snippet, so your labels stay valid when you re-chunk. This is the number to track over time — not the synthetic one.

Upgrades — add each only when the metric says so

1 · Persist
Chroma writes the HNSW index to disk. Reopen, don't re-embed.
2 · Rerank
FlashRank cross-encoder reorders the top-30. The usual next win once recall caps.
3 · Route
Send each query to the right collection. Separate routing misses from retrieval misses.

1 · Persist — ChromaDB on disk

Chroma stores the HNSW index and documents under path automatically — no save() call. Reopen the same path next run and skip re-embedding. At millions of vectors, move to a hosted store (Qdrant, pgvector, Pinecone) — same idea (embed query → nearest neighbors), different API.

copy
RAG("docs", path="./rag_db").add(chunks)   # writes ./rag_db
r = RAG("docs", path="./rag_db")           # next run: reopens, no re-embedding

2 · Rerank — real cross-encoder, and prove it helped

copy
base = r.evaluate(k=5)
lift = r.evaluate(k=5, rerank=True)   # pull 30 from Chroma, FlashRank reorders
print(base["recall@k"], "→", lift["recall@k"])   # ship it ONLY if this goes up

r.answer("your question", rerank=True)

3 · Route across collections

copy
from rag import RAG, Router

billing = RAG("billing", "invoices, refunds, payment errors").add(billing_docs)
api     = RAG("api", "endpoints, auth, rate limits").add(api_docs)

bot = Router([billing, api])
bot.answer("why was my card declined?")   # → routed to billing, then answered
